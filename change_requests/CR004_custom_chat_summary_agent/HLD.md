# Title: Custom Chat Summary Agent
## Status: Draft

## Background

Marinara needs a first-class way for an agent to maintain chat summaries during generation. Earlier design work explored two parts of this problem:

- CR001 proposed explicit built-in tools for reading and appending chat summaries.
- CR003 proposed a generalized metadata I/O bridge so tools can read and patch chat metadata without depending on route-specific database handles.

This CR supersedes those designs for the chat-summary use case. It should stand alone as the design record for implementing a custom chat summary agent and the supporting tool, agent-management, and metadata-sync improvements.

## Problem Statement

Chat summaries are long-term conversation memory, but they are currently not managed through a dedicated, configurable agent workflow. Summary updates should be generated by an agent that has access to the current chat context and can update the persisted summary without leaking database concerns into tool implementations.

The current direction also needs to avoid brittle tool context plumbing. Built-in and custom tools should receive a stable execution context that exposes chat metadata through a read/write bridge rather than requiring individual tools to know about `db`, `chatId`, or route internals.

## Goals

- Add a configurable custom chat summary agent that can maintain summaries as part of chat generation.
- Provide explicit summary-oriented tool behavior for reading the current summary and writing summary updates.
- Introduce or refine a generalized tool metadata I/O context so metadata-aware tools are decoupled from database and route internals.
- Ensure summary updates triggered by agents are persisted consistently.
- Ensure the client refreshes summary-dependent UI when background summary metadata changes.
- Improve agent management so the summary agent can be enabled, configured, and reasoned about consistently with other agents.
- Keep the design compatible with the upstream PR workflow by avoiding local-only artifacts in implementation commits.

## Non-Goals

- Reworking the entire agent system beyond what is needed for summary-agent support.
- Introducing a full memory subsystem for all metadata types.
- Redesigning chat storage or changing the persisted chat metadata format unless required for summary updates.
- Shipping CR001 or CR003 as separate implementation lines.

## Proposed Solution

### 1. Custom Chat Summary Agent

Add a summary-focused agent mode or built-in agent configuration that can run during an appropriate generation phase, likely post-processing after the assistant response is available. The agent should receive enough context to decide whether the summary needs an update, including the prior summary, recent messages, and the generated response.

The agent should produce structured summary updates rather than free-form side effects where practical. The implementation should make it clear when the agent is disabled, skipped, or unable to update the summary.

### 2. Summary Tools

Provide summary-oriented tool capabilities:

- `read_chat_summary`: returns the current persisted summary from chat metadata.
- `append_chat_summary` or equivalent summary-update behavior: accepts new summary content or a patch and requests a metadata update.

The final API can be adjusted during design iteration, but tool behavior should not depend directly on route database handles.

### 3. Generalized Tool Metadata I/O

Update tool execution context to expose chat metadata through stable read/write primitives:

```ts
chatMeta?: Record<string, unknown>;
onUpdateMetadata?: (patch: Record<string, unknown>) => Promise<void>;
```

Tools should read from `chatMeta` and write by calling `onUpdateMetadata`. Route-level orchestration remains responsible for persistence, local metadata merging, and client notification.

### 4. Route Orchestration

Generation routes should construct a shared tool context for the main assistant and all agents. The context should:

- include parsed chat metadata for fast reads;
- persist metadata patches through existing chat storage;
- merge patches into the in-memory metadata object used during the request;
- emit a metadata-change signal to the client stream when a patch is applied.

This should also reduce duplicated tool-preparation logic where possible.

### 5. Client Metadata Refresh

The client should respond to a metadata patch event, such as `metadata_patch`, by invalidating or refreshing the relevant chat detail query. Summary popovers or other summary UI should update without requiring a manual page refresh.

### 6. Agent Management Improvements

Agent management should expose the summary agent in a way that fits existing agent configuration patterns. The design should clarify:

- whether the summary agent is a built-in agent type, a template, or a normal configurable agent with special defaults;
- how it is enabled per chat or globally;
- which tools it is allowed to use;
- how failures are reported or logged.

## Risks and Open Questions

- The right phase for the summary agent needs validation. Post-processing is the likely default, but streaming behavior may affect when metadata can be safely patched.
- Summary update format needs refinement: append-only text, replacement summary, structured patch, or a combination.
- Automatic summary updates may need guardrails to avoid noisy or low-value summary growth.
- Existing agent tool execution paths may not all receive equivalent context today.
- Client invalidation must avoid excessive refetching during long streams.

## Validation

- `pnpm check` passes.
- Manual test: enable the summary agent, generate a chat response, and verify the persisted summary updates.
- Manual test: verify summary UI refreshes when the agent updates metadata.
- Manual test: verify tools still execute for normal agents and do not regress existing custom-tool behavior.
- Manual test: verify disabled summary-agent configuration has no side effects.
